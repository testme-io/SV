Product Understanding

Based on the case study materials — what I understand about CoPilotMD before our first conversation.

What the product does

CoPilotMD NV-Sight is an AI-powered assistant for neuro-endovascular procedures. It analyzes angiographic imaging in real-time while a physician performs an intervention — detecting vessel anomalies and ischemic areas, mapping them to brain functional regions, and helping the physician decide whether and how to treat.

Key clinical context Today this decision relies entirely on human interpretation under time pressure. A missed occlusion during a stroke procedure can be fatal. The system is advisory — it augments the physician, not replaces them.

Four modules — MVP scope

Module 1 — DETECT

Vessel Anomaly Detection

Real-time detection of occlusions and ischemic areas during interventional procedures. Accuracy >80%, within 12mm of ground truth.

Running live pilot @ Sheba
Module 2 — UNDERSTAND

Functional Brain Mapping

Correlates vessel anomaly with affected brain region using DSA + CTA + Atlas registration. Helps physician decide whether to treat.

Feasibility stage
Module 3 — INSIGHT

Navigation Assistance

Turn-by-turn catheter navigation to the occlusion site.

After MVP
Module 4 — REPORT

Automatic Reporting

Streamlined procedure documentation and workflow reporting.

After MVP

System architecture

Input: X-Ray Angio + PACS → CoPilotMD Workstation (Receiver → Data Organization → AI Classifier) → Output to INR Display. Currently running on a laptop within the hospital's network domain. No hardware changes required at the hospital side.

Clinical data available

Retrospective datasets

🇮🇱 Sheba Medical Center — 3,000+ patients, 600 stroke cases (pilot running)

🇺🇸 Mount Sinai, NY — 500 patients (250 stroke + 250 elective)

🇺🇸 St. Vincent Ascension, Indiana — 400 patients (200 stroke + 200 elective)

Upcoming studies

Retrospective: 1–2 additional US sites (~400–500 patients/site), expected in 3 months

Prospective observational: 2–3 US sites, 6–12 months timeline

QA status today

Per the case study: "Quality system setup from scratch." No formal QA process exists. The product is live at Sheba without documented testing, test plans, or quality gates.

Risks Identified

Based on the case study analysis — before our first conversation. Each risk has an expected answer and what it means for the QA plan.

🔴 Critical

Critical Training = Validation Dataset

The reported 80% accuracy may have been measured on the same data the model was trained on. If true — this number is not validated and cannot be used in an FDA submission.

Why it matters: FDA requires an independent validation set. US retrospective data (Mount Sinai, St. Vincent) must serve this role — and we need to confirm the model hasn't seen it during training.

Expected answer from CTO: Likely partial overlap. This is typical for early-stage startups. Not a catastrophe — but a gap that must be closed before submission.
Critical System in Clinic Without QA

Sheba pilot is live with real patients and no formal testing documentation. Any incident — even minor — creates regulatory and reputational exposure.

First action: Baseline audit. Document current system state, capture all known issues from the pilot, create a formal issue log from informal feedback.

Expected answer from CTO: "We know — that's why we're hiring." The goal is to get this acknowledged and move immediately to baseline documentation.

🟡 High

High Module 2 on Feasibility

MVP requires both Module 1 and Module 2. Module 2 is still proving technical feasibility — image registration between DSA, CTA and brain atlas. If it doesn't stabilize on time, the entire MVP timeline shifts.

Expected answer from CTO: No firm timeline. This is the honest answer for a feasibility-stage component. Plan must treat Module 2 as a milestone-gated dependency, not a fixed date.
High Laptop as Production Hardware

The system runs on a laptop in the hospital domain. Overheating, network loss, or freeze during a procedure is a real failure scenario. This is not a stable production environment for a medical device.

Expected answer from CTO: Temporary solution for pilot. Need to confirm target architecture for US commercialization — dedicated on-premise server is most likely.
High Each Hospital = New Integration

"No hardware changes" is a marketing statement. Each US site has different PACS, different angiography systems, different network configurations. Without a site acceptance testing protocol, every new connection is a risk.

Expected answer from CTO: Tested at Sheba, possibly 1–2 others. A formal compatibility matrix and site acceptance protocol doesn't exist yet.

🔵 Medium

Medium Ground Truth Annotation Status

Validation requires clinically annotated data — a radiologist or neurologist confirming what the ground truth is for each case. Without completed annotation, validation cannot start.

Expected answer from CTO: Partially annotated. This may be the critical path bottleneck for the entire validation plan.
Medium No Regulatory Consultant

No confirmed FDA or regulatory advisor on the team. Without one, QA documentation may be built in the wrong format or missing key requirements for submission.

Expected answer from CTO: Not yet engaged. Engaging a regulatory consultant should be part of the 30-day plan.
Medium IRB Approval & Data De-identification

Using patient imaging data requires IRB approval and proper de-identification per HIPAA standards. This is a compliance prerequisite for FDA submission.

Expected answer from CTO: Should be in place — hospitals wouldn't participate otherwise. But needs formal confirmation and documentation.

What I'll Need

To build a QA plan that actually works — these are the resources, access, and people required from day one.

People

🩺 Clinical Expert

Single source of truth for clinical correctness. Who is the physician/radiologist who approves that a functional brain mapping output is clinically correct — not just algorithmically accurate? This person is essential for validation.

⚖️ Regulatory Consultant

They define what FDA needs in the submission package. I build the QA system that produces it. Without this person, there's a risk of building correct documents in the wrong format.

👨‍💻 Engineering Access

Direct access to developers to understand the pipeline architecture — Receiver, Organization, AI Classifier, Output. Need to understand failure modes at each stage.

👶 Junior QA (Variant C only)

To run parallel tracks — operational QA process while I focus on regulatory. Even part-time for 3 months would significantly accelerate the plan.

Access to the datasets from all three sites — Sheba, Mount Sinai, St. Vincent. With annotation status and IRB documentation confirmed.

Jira or equivalent — configured for QA workflows, severity/priority matrix, and reporting. If already exists, needs QA-specific configuration.

Everything that exists — product requirements, architectural diagrams, clinical study protocols, IRB approvals, any informal test notes. Starting point for remediation.
